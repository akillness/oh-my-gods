# Context

## Workflow Context

- The recurring loop has already closed `debugging`,
  `performance-optimization`, `monitoring-observability`, `log-analysis`,
  `environment-setup`, `user-guide-writing`, `data-analysis`,
  `pattern-detection`, `task-planning`, and `task-estimation`.
- Repo-wide spec validation still passes, so the next value is not schema
  repair. The value is sharper routing, progressive-disclosure packaging, and
  trigger checks for still-generic workflow skills.
- The strongest remaining under-packaged PM pair is now `standup-meeting` and
  `sprint-retrospective`.

## Affected Users

- Maintainers who need one bounded recurring lane at a time without reopening
  merged work
- Agents that need daily coordination help without drifting into planning,
  estimation, or retrospective behavior
- Reviewers who need objective trigger and output-shape checks before deeper
  optimization loops

## Current Workarounds

- Rely on a generic standup template that still reads like boilerplate scrum
  advice instead of a routing-aware coordination workflow
- Manually infer when a request should move to `task-planning`,
  `task-estimation`, or `sprint-retrospective`
- Skip local references and evals, which makes future rewrites harder to judge

## Adjacent Problems

- `standup-meeting` overlaps with `task-planning` and `task-estimation` unless
  its routing boundaries are explicit
- Single-file workflow skills without support files drift faster than compact
  entrypoints with references and evals
- Starting `skill-autoresearch` before the basic package is in place would
  freeze a weak baseline

## User Voices

- Current ecosystem guidance still rewards portable skill directories with
  reusable references, evals, and compact entrypoints over oversized inline
  templates
- Daily coordination surfaces are especially sensitive to route-out quality
  because they sit next to planning, estimation, and retrospective workflows
