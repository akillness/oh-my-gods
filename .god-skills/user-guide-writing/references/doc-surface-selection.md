# User-Guide Surface Selection

Use this reference when the user wants "docs" but the correct document type is
not obvious yet.

## Choose the smallest document that matches the user goal

Prefer:

- getting started: first successful setup or first completed task
- how-to: one repeatable task with minimal explanation
- tutorial: guided learning with explanation plus practice
- FAQ: short answers to recurring questions
- troubleshooting: recover from a known error or confusion point

Do not bundle multiple shapes into one article unless the user explicitly wants
a hub page or learning path.

## Audience first

Capture who the document serves before drafting:

- end user
- workspace or product admin
- support operator
- customer success or training lead

The same feature may need different docs for different audiences.

## Route-outs that keep this skill clean

Route away when the primary deliverable is:

- internal specs, ADRs, migration plans, or runbooks: `technical-writing`
- API reference, webhook docs, SDK setup, or integration examples: `api-documentation`
- release notes, patch notes, or version-history summaries: `changelog-maintenance`

## Checklist before drafting

- one audience
- one main task
- one clear starting point
- one success condition
- only the most likely failure paths
